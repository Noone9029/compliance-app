# Client-Ready V1 Completion Plan

## Summary
- Baseline is currently green: `pnpm --filter @daftar/api test`, `pnpm --filter @daftar/web test`, and `pnpm typecheck` all pass.
- Confirmed core gaps are architectural, not just cosmetic: org/member administration is missing, password reset still exposes raw links, `/v1/auth/debug` is still present, the worker does not process jobs, ZATCA is still an internal success-path simulator, connector and billing actions still imply live integrations that do not exist, and several finance surfaces still describe or compute values as “estimated”.
- Delivery order will be vertical slices in this order: `Auth + Team Admin`, `ZATCA + Queue + Immutability`, `Attachments + Payments + Document Hardening`, `Reports + Dashboard Consistency`, `Connector/Billing Honesty + Client-Facing Cleanup`.

## Implementation Changes
- **High risk: Auth, organization, and membership readiness.** Add real org-scoped team administration using existing `platform.membership.read/manage` permissions: list members and invitations, create/send invite, resend/revoke invite, change role, disable/re-enable membership, and surface this under a new settings area such as `Team & Access`. Remove customer-visible debug behavior, keep session/org-switch redirects stable, and refresh capability snapshots immediately after role changes.
- **High risk: Invite and password-reset delivery.** Replace the current raw-link UX with real outbound email delivery wired through existing template/settings infrastructure. Public reset/request flows remain available, but they must never reveal tokens in the UI. If delivery fails, return a neutral success message to the user, record audit/error state server-side, and expose operational visibility only to authorized staff.
- **Highest risk: ZATCA completion.** Extend the compliance domain beyond the current `ComplianceDocument`/`ZatcaSubmission` success path to include onboarding/device records, certificate or CSID lifecycle state, submission queue items, transport attempts, retry history, failure classification, and event timeline. Separate clearance and reporting flows: standard tax invoices and associated notes must clear before “share/download” is considered final; simplified invoices and notes must be stamped locally and reported asynchronously within the 24-hour window. Enforce per-unit invoice counter continuity and previous-hash chaining, keep issued documents immutable, and replace the disabled e-invoice “Details” action with a real detail/timeline surface.
- **High risk: Worker-backed compliance processing.** Turn `apps/worker` into a real processor for compliance queue jobs, retry backoff, dead-letter or terminal failure handling, onboarding renew/revoke tasks, and operator-triggered retries. API/controller responses should enqueue work instead of faking immediate success, while UI surfaces show `pending`, `accepted`, `accepted with warnings`, `rejected`, and `retry scheduled` states from persisted attempts.
- **High risk: Standards-aware document generation.** Add a dedicated XML/archive service boundary that produces the canonical invoice payload, UUID, QR TLV payload, previous hash, external submission identifiers, and human-readable output from the same source model. Support XML download and compliant archive storage now; only advertise PDF/A-3 when the embedded-XML and signature path is genuinely complete, otherwise keep PDF/A-3 behind an internal interface and do not market it as done.
- **Medium/high risk: Attachments and issued-document hardening.** Replace JSON base64 uploads in customer flows with real binary upload handling against the existing S3-compatible storage abstraction, with local MinIO/dev fallback. Add download and delete rules, storage cleanup, and attachment audit entries. Issued/cleared/reported documents must not allow mutation or attachment removal if that would violate archival/compliance posture.
- **Medium/high risk: Payments, bank accounts, and downstream consistency.** Add `bankAccountId` to invoice and bill payment records so cash movement is tied to a real selected bank account instead of being assigned to the primary account heuristically. Update payment forms, DTOs, services, audit logs, and report aggregation accordingly.
- **Medium risk: Sales, purchases, quotes, inventory, and journals.** Keep the current document surface, but harden state transitions, idempotent quote-to-invoice conversion, repeating-document run guards, immutable-issued posture, and attachment/history visibility across invoices, bills, and quotes. Keep inventory quantity synchronization deterministic across imports, adjustments, and document status changes; keep manual journals balanced and make sure they remain part of report derivation. Do not add a speculative costing engine beyond the current product shape.
- **Medium risk: Reports and charts.** Replace current “estimated” or partial logic with a single posting/derived-ledger layer that uses posted sales, purchases, bank-linked payments, manual journals, fixed-asset depreciation, and existing inventory-linked document effects. Upgrade every visible report route to full filtered data rather than top-N/dashboard approximations: executive summary, profit and loss, balance sheet, trial balance, bank summary, receivables/payables detail, sales tax, ZATCA tax/reporting, reported documents, and contact transactions. Keep export only where it is backed by the live filtered dataset.
- **Medium risk: Connector and billing posture, per chosen scope.** Because the chosen end-state is `ZATCA first`, do not ship fake live connector OAuth/sync or live Stripe upgrade/manage actions in this pass. Convert those surfaces into honest readiness and record pages: current config, history, logs, stored state, and clear “not yet connected” messaging, with write actions hidden unless real backend transport exists. This preserves client-facing honesty without inventing unsupported integrations.
- **Low/medium risk: Client-facing cleanup.** Remove week/internal/demo wording from UI, seeded records, compliance messages, and subscription/connectors copy. Keep demo tenants professional and neutral. Leave only `HR & Payroll`, `Task Management`, `Applications`, and `List Tracking` as shell-level modules.

## Public Interfaces
- Add membership-management APIs and web contracts for member list, invitation lifecycle, role updates, and membership disable/restore.
- Add outbound email service configuration and template usage contracts for invitation and password-reset delivery.
- Change payment DTOs and records to include `bankAccountId`.
- Change attachment upload/download contracts to real binary upload plus delete support.
- Expand compliance DTOs returned to invoice detail, compliance overview, and e-invoice integration pages to include onboarding state, device/certificate state, submission queue state, attempts, retry data, and immutable issuance metadata.
- Expand report payloads where needed so bank summary and cash movement expose real `opening`, `inflow`, `outflow`, `closing`, and any `unassigned` movement explicitly instead of implicit primary-account assignment.

## Test Plan
- Keep the current green baseline as the pre-change gate: API tests, web tests, and typecheck.
- Add unit coverage for UUID generation, previous-hash chaining, QR TLV payloads, document classification, retry/backoff policy, attachment storage rules, payment-to-bank aggregation, and journal balancing.
- Add API coverage for member admin and invitations, reset email delivery behavior, tenant isolation on all new mutations, document immutability after issue/clear/report, attachment delete/download rules, compliance onboarding/queue/retry flows, bank-linked payment mutations, and hidden connector/billing write actions.
- Add integration coverage for worker processors, offline or failed ZATCA transport with retry history, certificate lifecycle transitions, object-storage cleanup, and seed verification for the cleaned client-facing dataset.
- Add UI coverage for `Team & Access`, auth flows without raw links, e-invoice detail/timeline/retry states, report filters/exports/no-data states, and the removal of misleading connector/Stripe controls.

## Assumptions and Defaults
- Chosen scope: `ZATCA first`. Real ZATCA transport is in scope now; connector OAuth/live sync and live Stripe checkout/portal are not. Those surfaces must become honest/readiness-only until real transport exists.
- Chosen auth posture: `Real email`. Invitation and password-reset delivery will be provider-backed; provider credentials remain an external setup dependency.
- Storage default: keep the S3-compatible abstraction and use local MinIO/dev fallback; do not move files into database blobs.
- Navigation default: add one new settings area for team/member administration; otherwise preserve the current shell and tenant route structure.
- Compliance default: if full PDF/A-3 generation/signing cannot meet the actual standard within the current stack, ship XML plus human-readable PDF without claiming PDF/A-3 support.
